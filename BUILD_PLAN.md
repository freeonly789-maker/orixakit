## Executive Summary

This document outlines the complete implementation plan for **Orixakit**, a production-ready shadcn-compatible component registry. The registry exposes installable components through three independent surfaces: the shadcn CLI, an MCP (Model Context Protocol) server for AI assistants, and a Skills system for AI coding tools.

The project is designed to be modular, allowing users to leverage any of the three surfaces independently or simultaneously. All components follow shadcn conventions and integrate seamlessly with existing Next.js applications.

## Project Architecture

### Core Components

Orixakit consists of four primary subsystems:

| Subsystem | Purpose | Users |
|-----------|---------|-------|
| **Registry API** | Serves component metadata and source files via HTTP | shadcn CLI, external tools |
| **MCP Server** | Exposes tools for browsing, searching, and installing components | AI assistants (Claude, Cursor, Windsurf) |
| **Skills System** | Provides markdown-based usage guides for AI tools | Manus AI, other AI coding agents |
| **Homepage** | Displays all available components and provides documentation | Developers, end users |

## Implementation Phases

### Phase 1: Project Initialization

**Objective:** Set up the Next.js project with shadcn dependencies and configuration.

**Steps:**

1. Initialize Next.js 15 project with TypeScript, Tailwind CSS, and ESLint
2. Install shadcn CLI and initialize with New York style and Slate base color
3. Install core shadcn components: `button`, `input`, `card`, `label`
4. Install MCP SDK: `@modelcontextprotocol/sdk` and `zod`
5. Install development tools: `tsx` for running TypeScript files

**Configuration Files to Create:**

- **components.json:** Configure shadcn with New York style, RSC enabled, and registry namespace for `@orixakit`
- **package.json scripts:** Add `registry:build`, `registry:dev`, `mcp:start`, `mcp:dev` commands
- **registry.json:** Master registry definition with all component metadata

**Deliverable:** Fully configured Next.js project with shadcn CLI ready to build registry items.

### Phase 3: Registry API Implementation

**Objective:** Expose components via HTTP endpoints for shadcn CLI consumption.

**Dynamic Route Handler:** `app/r/[name]/route.ts`

This endpoint serves individual component metadata and source files in shadcn-compatible JSON format:

- **Endpoint:** `GET /r/{name}.json`
- **Response:** Complete component metadata including files, dependencies, and registry dependencies
- **Headers:** CORS enabled, cache control set to 1 hour
- **Error Handling:** Returns 404 with descriptive message if component not found

**Registry Index Endpoint:** `app/registry.json/route.ts`

Exposes the master registry definition for tools that need to browse all available components:

- **Endpoint:** `GET /registry.json`
- **Response:** Complete registry.json with all component metadata
- **Headers:** CORS enabled for cross-origin access

**Deliverable:** Fully functional HTTP API compatible with shadcn CLI and external tools.

### Phase 4: MCP Server Implementation

**Objective:** Build an MCP server that exposes registry data to AI assistants.

**Server Entry Point:** `mcp/server.ts`

The MCP server implements four tools that AI assistants can invoke:

#### Tool 1: list_orixakit_components

Lists all available components, optionally filtered by category (ui, blocks, hooks, ai).

**Parameters:**
- `category` (optional): Filter by category; defaults to "all"

**Response:** Formatted text list of components with descriptions

#### Tool 2: get_orixakit_component

Fetches full metadata and file contents for a specific component.

**Parameters:**
- `name` (required): Component name (e.g., "orix-button")

**Response:** JSON object containing metadata and built payload

#### Tool 3: search_orixakit_components

Performs full-text search across component names, titles, descriptions, and categories.

**Parameters:**
- `query` (required): Search keyword

**Response:** Filtered list of matching components

#### Tool 4: get_install_command

Returns the exact CLI command to install one or more components.

**Parameters:**
- `names` (required): Array of component names

**Response:** Installation instructions for both namespace and direct URL methods

**Integration Instructions:**

The MCP server can be connected to AI tools via configuration files:

- **Claude Code:** Add to `~/.claude/claude_desktop_config.json`
- **Cursor/Windsurf:** Add to `.cursor/mcp.json` or `.windsurf/mcp.json`

**Deliverable:** Fully functional MCP server with comprehensive tool implementations and integration documentation.

## Registry Configuration

### registry.json Structure

The master registry definition contains all component metadata:

```json
{
  "$schema": "https://ui.shadcn.com/schema/registry.json",
  "name": "orixakit",
  "homepage": "https://orixakit.dev",
  "items": [
    {
      "name": "orix-button",
      "type": "registry:ui",
      "title": "Orix Button",
      "description": "Extended button with loading state, icon support, and size variants.",
      "registryDependencies": ["button"],
      "files": [
        {
          "path": "registry/new-york/ui/orix-button/orix-button.tsx",
          "type": "registry:ui"
        }
      ],
      "categories": ["ui", "primitives"]
    }
    // ... additional items
  ]
}
```

**Key Fields:**

- **name:** Unique identifier for the component
- **type:** Registry type (registry:ui, registry:block, registry:hook, registry:component)
- **title:** Display name for the component
- **description:** Brief description of functionality
- **registryDependencies:** Other registry items this component depends on
- **dependencies:** NPM packages to install
- **files:** Array of files to install with their types and target paths
- **categories:** Tags for filtering and organization

### components.json Configuration

```json
{
  "$schema": "https://ui.shadcn.com/schema.json",
  "style": "new-york",
  "rsc": true,
  "tsx": true,
  "tailwind": {
    "config": "tailwind.config.ts",
    "css": "app/globals.css",
    "baseColor": "slate",
    "cssVariables": true
  },
  "aliases": {
    "components": "@/components",
    "utils": "@/lib/utils",
    "ui": "@/components/ui",
    "lib": "@/lib",
    "hooks": "@/hooks"
  },
  "registries": {
    "@orixakit": "https://orixakit.dev/r/{name}.json"
  }
}
```

**Local Development Note:** During development, replace `https://orixakit.dev/r/{name}.json` with `http://localhost:3000/r/{name}.json` in components.json.

## Maintenance and Updates

### Adding New Components

1. Create component source file in `registry/new-york/{category}/{name}/`
2. Add entry to `registry.json` with complete metadata
3. Run `npm run registry:build` to generate JSON artifacts
4. Update relevant SKILL.md file if adding to existing category
5. Test with `npx shadcn@latest add http://localhost:3000/r/{name}.json`

### Updating Existing Components

1. Modify component source file
2. Update metadata in `registry.json` if needed
3. Run `npm run registry:build`
4. Test installation and functionality

### Version Management

- Maintain semantic versioning in package.json
- Document breaking changes in CHANGELOG
